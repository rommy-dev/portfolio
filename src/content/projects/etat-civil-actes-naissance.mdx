## Contexte du projet

Dans le cadre de mon stage au **Ministère de l'Intérieur et de la Décentralisation** d'Antananarivo, j'ai été missionné pour concevoir et développer une application de saisie rétroactive des actes d'état civil — en particulier les actes de **naissance**.

Ce type de projet répond à un enjeu critique : des milliers d'actes de naissance établis en dehors des délais légaux n'ont jamais été intégrés dans les systèmes informatisés existants. L'objectif est de les numériser de manière sécurisée tout en garantissant l'intégrité et la traçabilité des données.

> Il s'agit de mon premier projet en environnement gouvernemental — une expérience qui m'a confronté à des exigences de sécurité, de confidentialité et de rigueur bien au-delà d'un projet classique.

---

## Fonctionnalités principales

- **Saisie guidée** des informations de naissance : identité, date, lieu, témoins, officiant
- **Validation métier** stricte avec règles personnalisées selon la législation malgache
- **Gestion des rôles** : agents de saisie, superviseurs, administrateurs
- **Historique et traçabilité** de toutes les modifications sur chaque enregistrement
- **Recherche avancée** par nom, date, commune, arrondissement
- **Export PDF** des actes formatés selon les modèles officiels
- **Tableau de bord** de suivi pour les superviseurs (volume saisi, taux d'erreur, etc.)

## Architecture technique

### Backend — Symfony 7 + API Platform

Le backend est construit sur **Symfony 7** avec **API Platform** pour exposer les ressources en REST. Ce choix garantit :

- Une structure claire en entités Doctrine mappées sur MySQL
- Une documentation API auto-générée (OpenAPI / Swagger)
- Un système d'authentification par **JWT** avec gestion fine des permissions par rôle
- Des validations robustes grâce aux contraintes Symfony Validator

```bash
# Structure simplifiée du backend
src/
├── Entity/           # ActeNaissance, Personne, Commune...
├── ApiResource/      # Décorateurs API Platform
├── Security/         # Voters, JWT config
├── Repository/       # Requêtes Doctrine personnalisées
└── EventSubscriber/  # Logs d'audit automatiques
```

### Frontend — React + TypeScript

L'interface est développée en **React** avec **TypeScript** pour garantir la robustesse des formulaires complexes. Les choix techniques incluent :

- **React Hook Form** pour la gestion des formulaires multi-étapes
- **Zod** pour la validation côté client synchronisée avec les contraintes backend
- **TanStack Query** pour la synchronisation des données avec l'API

### Base de données — MySQL

La modélisation a été un défi important : les données d'état civil ont des relations complexes (personnes impliquées dans plusieurs actes, adresses historiques, référentiels géographiques malgaches).

| Table | Rôle |
|---|---|
| `acte_naissance` | Enregistrement principal |
| `personne` | Entité partagée (né, parents, témoins) |
| `commune` | Référentiel géographique |
| `utilisateur` | Agents avec rôles |
| `audit_log` | Traçabilité complète |

## Défis techniques rencontrés

### Sécurité des données sensibles

Travailler avec des données d'état civil implique des contraintes strictes. J'ai mis en place :

- Chiffrement des champs sensibles en base via **Doctrine Encrypted Bundle**
- Logs d'audit immuables sur toutes les modifications
- Rate limiting sur les endpoints critiques
- Validation stricte côté serveur, indépendante du frontend

### Gestion des données manquantes

Les actes anciens sont souvent incomplets : date approximative, prénoms inconnus, commune incertaine. Il a fallu concevoir un modèle de données flexible avec des champs optionnels bien documentés et un système de *flags* pour indiquer le niveau de complétude d'un acte.

### Cohérence des référentiels géographiques

Madagascar possède une organisation administrative spécifique (régions, districts, communes, fokontany). J'ai construit un référentiel hiérarchique importé depuis les données officielles et utilisé comme source unique de vérité dans toute l'application.
